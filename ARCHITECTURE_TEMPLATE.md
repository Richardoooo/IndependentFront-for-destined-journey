## 二、后端需求

### 2.1 Prompt 结构

<!-- 发给 AI 的 prompt 应该包含哪些部分？顺序？ -->
需要保持高命中缓存，我推荐多agent模式下加上固定提示词在前，会变的提示词在后。
然后不同的agent享有不同的命中缓存，保持省钱最大化
ai至少需要知道世界观的基础设定+自己的职责

### 2.2 Agent 分工

<!-- 需要哪些 Agent？各自负责什么？怎么调度？ -->
目前的构想（按输入流程分）

用户输入 -> 先调用记忆召回agent，分析上轮和用户这轮输入的内容可能触发之前的什么记忆，设置里可以设置一个记忆召回上限，召回的是详细的记忆 -> 剧情规划agent，这个其实可以和记忆召回并行，主要为判断此时此刻需要触发什么事件，返回一个按照时间点可能触发的事件 -> 正文ai，基于记忆和剧情规划，配合思维链（参考命定之诗Kemini5-3.8.json相关内容）推进正文 -> 变量更新agent，主要处理时间，地点，在场人物，新闻等内容 -> 角色更新agent，在变量更新后，并行多处理，一个角色发配给一个agent，负责更新该角色的所有内容（例如现在在做什么，以及技能，背包，血量各种等等，包括主角的角色）
-> 记忆agent再次调用，配合剧情规划的prompt总结这次正文做了什么（这个可以并行吗？） -> 记忆规划再次调用，查看正文里是否有内容会"改变世界线"，然后对规划好的剧情进行修改.

职责详解:
记忆召回agent：提示词应该包含所有的记忆（按照条目计），然后按照从近到远的样子给需要召回的记忆排好序，返回一串json列表
记忆相关：记忆应该包含以下内容：记忆编号"MEM00XXX"，时间跨度"0001/05/24/23:11 - 0001/05/24/23:25"，正文内容（主角做了什么，附近发生了什么，不少于200字），暗线内容（不在场的配角做了什么）

剧情规划agent：
剧情规划应该在设置中可开可不开，开启后，具体描述应该是这样的：
首先，剧情应该以各种事件为起始点，拥有持续时间,我设想的数据结构应该类似于嵌套式，例如剧情1（起因经过结果,涉及人物，事件类型(天灾，人祸)）其中嵌套各种 小事件（起因经过结果），如果不够还能在套一层，然后生成剧情的时候要优先考虑合理性，以及起伏转折等。
大事件：起因经过结果，时间地点人物，事件类型，嵌套的小事件
小事件：起因经过结果，时间地点人物，促成了大事件下的哪些小事件，可能存在的线索。

上面是初始化的内容，或者可以在游戏里的每隔一年主动初始化一次，每次生成一些小剧情。用户可以选择剧情类型，主打探索/感情互动/天灾等等等等

然后平常调用时，判断用户是否可能参与进剧情，返回小事件（自行判断证据是否足够推断出大事件，否则返回小事件），这里可能还需要讨论具体返回的格式？

然后是剧情规划最后一次调用，判断主角是否做了什么事情导致世界线变动，这种情况就可以把主角做的事新加入一个小事件，然后适当修改时间线后的事件。

正文ai：
处理正文，先复盘记忆，复盘剧情规划注入内容，复盘角色人设，复盘世界设定（经济，等级等内容），复盘文风，然后思考正文怎么出。
**重要**：需要做一个工具，把上次正文中不合规的语句（比如，我不想让ai输出不是....而是.....），那我这次注入的时候就需要返回一个包含违规语句的列表，让ai不要这么写。
我目前想把物品制作和战斗做成那种函数调用的形式，让后端判断成功与否，避免口胡，具体战斗你可以看一下命定之诗与黄昏之歌v4.2 绿灯命中微调版 (1).json里的战斗协议

变量更新agent：处理时间，地点，在场人物是否在场，离场，新闻内容，具体可以参考json里的mvu update

角色更新agent：先判断时间跨度，判断角色人设，然后看正文，更新各种内容。
